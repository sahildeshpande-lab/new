# Reporting Backend Documentation

## Purpose

This backend converts natural-language reporting questions into MySQL reports.

The core flow is:

1. Connect to the configured MySQL database.
2. Read live schema details from `information_schema`.
3. Load business metadata from `reporting_metadata.json`.
4. Send the user question, schema, metadata, filters, and row limit to Gemini.
5. Gemini generates a single MySQL `SELECT` query and named parameters.
6. The backend validates the SQL before execution.
7. The backend fetches rows from MySQL.
8. Gemini generates the final written report summary from the returned rows.

V1 supports only these report families:

- Attendance Report
- Timesheet Report
- Team Lead Timesheet Report
- HR Timesheet Report

Frontend/Next.js is not part of this backend phase.

## Environment

Create `.env` in the project root:

```env
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_DATABASE=devita_project_management
MYSQL_USER=root
MYSQL_PASSWORD=
MYSQL_SSL_DISABLED=true

GEMINI_API_KEY=your_gemini_key
GEMINI_BASE_URL=https://generativelanguage.googleapis.com/v1beta
GEMINI_MODEL=gemini-2.5-flash-lite
LLM_TEMPERATURE=0.1
LLM_MAX_TOKENS=2500
```

Use the actual MySQL server host, not the phpMyAdmin URL. phpMyAdmin is only a web UI for MySQL.

## Endpoint Summary

| Endpoint | Method | Use Case |
| --- | --- | --- |
| `/health` | `GET` | Check if the API process is running. |
| `/llm/health` | `GET` | Check Gemini API key/model access without sending database rows. |
| `/schema/summary` | `GET` | Confirm database connectivity and inspect live MySQL tables/columns. |
| `/schema/effective-metadata` | `GET` | View the exact live-schema-reconciled metadata sent to Gemini. |
| `/reports` | `GET` | List supported V1 report types and filters from metadata. |
| `/reports/query` | `POST` | Main endpoint for natural-language report requests. |
| `/reports/{report_id}/run` | `POST` | Run a specific approved report with filters, still using Gemini to generate SQL. |

## When To Use What

Use `GET /health` when:

- You want to verify FastAPI is running.
- You are checking deployment or local server status.
- You do not need database or LLM validation.

Use `GET /schema/summary` when:

- You want to test MySQL credentials.
- You need to confirm which tables and columns exist in the live database.
- The LLM generated SQL fails because the actual schema differs from the docs.

Use `GET /schema/effective-metadata` when:

- You want to confirm Gemini is seeing the same columns shown in phpMyAdmin.
- You want to verify stale documented columns such as `attendances.check_in` were removed.
- You need to debug why Gemini selected a specific table or column.

Use `GET /llm/health` when:

- Gemini returns `400`, `403`, `404`, or model access errors.
- You want to confirm the API key can list models.
- You want to confirm `GEMINI_MODEL` is visible to your Google AI Studio project.
- You do not want to send database rows to Gemini during debugging.

Use `GET /reports` when:

- You want to see the four reports supported in V1.
- You want to inspect allowed filters before building a request.
- You need the exact `report_id` for `/reports/{report_id}/run`.

Use `POST /reports/query` when:

- The user asks a normal natural-language question.
- You want the LLM to choose the correct report type.
- Example: “Show attendance report for March 2026”.
- Example: “Give HR timesheet report for employee name Amit from 2026-03-01 to 2026-03-31”.

Use `POST /reports/{report_id}/run` when:

- Your app already knows the report type.
- You want to constrain Gemini to a specific report.
- You want a more predictable backend integration from a future frontend.
- Example report IDs:
  - `attendance_report`
  - `timesheet_report`
  - `team_lead_timesheet_report`
  - `hr_timesheet_report`

## Main Natural-Language Endpoint

`POST /reports/query`

Request:

```json
{
  "question": "show attendance report for March 2026",
  "filters": {},
  "format": "json",
  "limit": 50
}
```

Response:

```json
{
  "report_id": "attendance_report",
  "report_name": "Attendance Report",
  "sql_preview": "SELECT ... LIMIT :limit",
  "applied_filters": {
    "date_from": "2026-03-01",
    "date_to": "2026-03-31",
    "limit": 50
  },
  "columns": ["date", "employee_name", "status", "worked_hours"],
  "rows": [],
  "generated_report": "No attendance rows matched the selected period.",
  "warnings": [],
  "clarification": null
}
```

The SQL and column names shown above are examples. The actual SQL is generated by Gemini and then validated by the backend.

## Specific Report Endpoint

`POST /reports/{report_id}/run`

Use this when the caller already knows the report.

Request:

```json
{
  "question": "Generate timesheet report for project 10 in March 2026",
  "filters": {
    "date_from": "2026-03-01",
    "date_to": "2026-03-31",
    "project_id": 10
  },
  "format": "json",
  "limit": 100
}
```

If `question` is omitted, the backend creates a basic question from the report name and filters. Gemini still generates the SQL.

## CSV Export

Set:

```json
{
  "format": "csv"
}
```

CSV responses return the fetched rows as `text/csv`. Use JSON when you need `sql_preview`, `warnings`, and `generated_report` in the same response.

## Safety Model

The backend does not directly execute arbitrary LLM output. Static JSON metadata is used for report business context; live MySQL `information_schema` is the source of truth for columns.

Before execution, SQL must pass these rules:

- Query must start with `SELECT`.
- Query must be a single statement.
- Query cannot contain write or DDL keywords.
- Query tables and columns must exist in the live MySQL database schema.
- Query must include `LIMIT :limit`.
- Every named SQL parameter must have a value.
- Reports, generated summaries, and test requests are returned to the caller only.
- The backend does not insert report output, generated summaries, test data, prompts, or SQL into MySQL.

The LLM prompt also instructs Gemini not to use table aliases. This keeps validation simple because columns must appear as `table.column`.

## Gemini API Errors

This error happens before SQL execution. No MySQL write has happened.

Common causes:

- `GEMINI_API_KEY` is missing, invalid, expired, or from the wrong Google AI Studio project.
- `GEMINI_MODEL` is not enabled for your key/project.
- The configured model name is wrong.
- Your network, VPN, firewall, or hosting IP is being rejected before the request reaches normal model execution.

Checks:

1. Restart `uvicorn` after editing `.env`.
2. Call `GET /llm/health`.
3. If `configured_model_available` is false, change `GEMINI_MODEL` to a model listed in `available_models_sample`.
4. Try from another network if `/llm/health` still returns 403.
5. In Google AI Studio, check API key access.

## Current Report Metadata

The curated metadata file is:

```text
reporting_metadata.json
```

It describes:

- database intent and read-only policy
- tables and columns
- relationships
- business meaning
- report aliases
- allowed filters
- SQL intent and computed fields

This file is the business knowledge layer for Gemini. If reports fail due to table or column mismatch, update this metadata after checking `/schema/summary`.

## Troubleshooting

If you see `Can't connect to MySQL server on '127.0.0.1'`, the app is using local MySQL defaults. Create `.env`, set the real MySQL host, and restart `uvicorn`.

If you see `GEMINI_API_KEY is missing`, add the key to `.env` and restart.

If Gemini generates SQL using a table or column not in metadata, the backend rejects it. Check `/schema/summary`, then update `reporting_metadata.json`.

If you need only raw rows and not a written report summary, keep using JSON and read the `rows` field. The backend still generates the narrative because V1 is designed as a report generator, not just a SQL runner.
